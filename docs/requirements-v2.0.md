## 一、系统定位

### 1.1 核心目标

**大幅减少企业文档编写和设计计算的工作量**，让 AI 承担 60% 以上的初稿工作，人工只需审阅、补充和修正。

### 1.2 五大功能目标

| # | 目标 | 说明 |
|---|------|------|
| 一 | 减少写文档和设计计算的工作量 | AI 生成大纲、报告正文、设计计算，覆盖 A/B/C 三档文档 |
| 二 | 让企业知道各阶段需要什么文档、怎么写 | 智能策划推荐 + 工作流导航 + 前置依赖检查 |
| 三 | 保证项目所有文档一致性 | 主数据统一提取 + 跨文档引用校验 + 指标联动更新 |
| 四 | 通过积累不断优化文档模型 | 企业知识复用 + 终稿对比学习 + 四层基线沉淀 |
| 五 | 多人协同高效率配合 | 审批流 + 并行编辑 + 任务分配 |

### 1.3 用户价值主张（面向采购决策者）

1. **节约人工**：量化 ROI，传统方式 vs 系统方式的人天对比
2. **减少审查困境**：自动合规检查 + 文档完整性保障 + 一致性校验
3. **越用越好**：系统积累企业经验，第二年比第一年省更多人工
4. **降低门槛**：新人跟着系统引导就能上手，从中学到文档写作知识

## 三、核心功能设计

### 模块一：文档质量标准体系

#### 3.1 五维评分模型

```
完成度分数 =
  章节完整性 × 30%    必填章节实有数 / 应有效
+ 内容覆盖度 × 25%    有效章节数 / 总章节数
+ 数据准确性 × 20%    主数据匹配数 / 主数据应有数
+ 格式规范性 × 15%    通过检查项 / 总检查项
+ 可读性     × 10%    基于段落长度、术语密度等

目标：初稿完成度 >= 60%
```

#### 3.2 四个维度目标

| 维度 | 目标 | 人工补足 |
|------|------|----------|
| 格式规范 | 85% | 页码、页眉、密级标识 |
| 章节结构 | 85% | 附录、引用文件列表 |
| 内容覆盖 | 60% | 项目特定数据、经验判断 |
| 数据准确 | 50% | 实测数据、企业经验值 |
| 逻辑一致性 | 65% | 跨文档引用校验 |

### 模块二：结构化 Prompt 生成体系

#### 4.1 新 Prompt 模板结构

```
一、角色与任务定义
二、必填章节清单（结构化定义，含字数约束）
三、主数据注入区（{{injected_context}}）
四、每章约束条件（必含字段、最小字数、禁止行为）
五、不确定性标记规范（标记格式）
六、质量元数据输出格式（每章末尾）
七、禁止事项（禁用占位词清单）
```

#### 4.2 核心改造点

**主数据强制注入**：
- 生成文档时先组装上下文，将已提取的主数据（产品名称、MTBF、环境条件等）显式注入 prompt
- 对缺失的数据，显式标记"缺数据，需用户补充"
- 锁定已知值，压缩 AI 的"瞎编空间"

**AI 主动标记不确定性**：

```xml
<!-- AI_META
章节: 可靠性要求
完成度: 55%
待核实:
  - MTBF值来源：使用了合同中的数据，建议与方案报告交叉验证
  - 振动条件：合同未明确，按GJB 150.16A通用要求填写，建议确认
确定项:
  - 可靠性定量指标
  - 任务剖面定义
缺项:
  - 维修性指标（主数据中无此信息）
-->
```

#### 4.3 分层生成流程

```
第一步：生成大纲（章节标题+摘要） -> 用户确认/调整
第二步：注入主数据，逐章生成正文  -> 每章标记不确定点
第三步：组装全文，统一格式        -> 自动检查 + 补充清单
```

### 模块三：对话式写作引擎

#### 5.1 核心理念

**从"一键生成"转变为"结构化引导对话"**：
- AI 不是输出完就闭嘴，而是在生成过程中不断和用户确认、纠偏
- 用户不是被动接收者，是主动参与者
- 对话本身就是教学（对新人）和决策沉淀（对系统）

#### 5.2 四阶段对话流程

```
阶段一：文档策划对话
  AI问 -> 用户答 -> AI确认
  目的：搞清楚要写什么、用什么数据、有什么约束

阶段二：大纲对话
  AI出大纲 -> 用户审 -> 用户改/确认 -> AI更新
  每章摘要 + 关键决策点标记

阶段三：逐章写作对话
  AI写一章 -> 标注不确定点 -> 用户纠正 -> AI重写
  -> 确认通过 -> 下一章

阶段四：全文审查对话
  AI自动检查 -> 输出审查报告 -> 用户逐项处理
  -> AI更新 -> 最终定稿
```

#### 5.3 AI 主动提问的三类关键节点

**节点 A：数据二选一或多选**
```
"MTBF验证方法未在合同中明确，可选：
 A) 定时截尾试验（常用，简单）
 B) 序贯试验（样本量灵活）
 C) 你自己输入"
```

**节点 B：缺数据**
```
"环境振动量级缺失，请提供或选择：
 A) 跳过，用标记
 B) 用GJB 150.16A默认值
 C) 上传振动谱文件"
```

**节点 C：章节确认**
```
"第4章已完成。完成度70%。2处待核实。
 是否通过？A)通过继续 B)修改本章 C)跳过暂存"
```

#### 5.4 对话数据沉淀

每次对话保存结构化决策记录：

```json
{
  "doc_id": "OUTLINE-2026-001",
  "decisions": [
    {
      "step": "可靠性要求.MTBF验证方法",
      "options": ["定时截尾","序贯试验","自定义"],
      "chosen": "定时截尾",
      "reason": "用户选择"
    },
    {
      "step": "第4章确认",
      "action": "修改",
      "before_score": 70,
      "after_score": 85,
      "corrections": ["补充了冗余设计具体措施"]
    }
  ],
  "final_score": 78
}
```

#### 5.5 前端交互形态

```
+------------------------------------------+
| 可靠性大纲 -- 对话式写作                    |
+------------------------------------------+
|                                          |
|  [AI] 大纲已规划8章，请确认。或者你希望调整    |
|       哪些章节？                           |
```

### 模块四：终稿对比学习系统

#### 6.1 五维差异分析

系统收到终版（评审通过后的最终版本）后，自动与初稿对比：

| 维度 | 分析内容 | 学习价值 |
|------|----------|----------|
| 结构层 | 章节增减、顺序调整、合并/拆分 | 企业偏好的文档结构 |
| 指标层 | 数值修正（MTBF 500->800 等） | 企业真实的指标区间 |
| 内容层 | 段落重写、描述补齐 | AI 哪些地方总是写不到位 |
| 格式层 | 编号、表格、术语修正 | 企业的格式规范 |
| 增删层 | 删了什么（冗余）、加了什么（遗漏） | AI 过度生成/遗漏的模式 |

#### 6.2 对比报告示例

```
【文档进化分析】可靠性大纲

初稿 V0.1 -> 终版 V1.0  总变化率：32%

结构调整 (1处)  新增"质量保证要求"章节
指标修正 (3处)  MTBF: 500->800, 可靠度: 0.95->0.98, 寿命: 空->15年
内容重写 (5段)  冗余设计、管理职责等
占位符消除 (12处)  10处填充，2处删除
新增内容 (+25%)  可靠性预计方法、HALT试验方案、FRACAS流程

-> 学习：该企业弹载设备需独立质保章节，指标偏高于GJB基线
```

#### 6.3 三层学习沉淀

```
第一层：项目级 -- 立即生效
  同一企业的下一个项目直接应用前一个项目的经验

第二层：企业级 -- 跨项目积累
  企业A：弹载设备 MTBF均值=650h，偏好定时截尾验证
  企业B：地面雷达 MTBF均值=1200h，偏好序贯试验

第三层：通用模型 -- 同类产品泛化
  弹载设备类：MTBF区间 300-1200h，必含工作项目清单
```

#### 7.3 依赖定义格式（结构化规则）


### 模块六：企业知识复用体系

#### 8.1 四层复用架构

```
第4层：项目级
  同平台系列型号，一键克隆全部文档架构

第3层：文档级
  同类文档的标准写法（企业模板，基于终稿优化）

第2层：段落/章节级
  高稳定性段落自动继承（引用文件、试验方法等）

第1层：决策/数据级
  企业基线参数（MTBF均值、偏好验证方法、术语习惯）
```

做完 3 个以上项目后，系统自动建立基线：

```
企业A . 弹载电子设备 基线（基于5个项目）

可靠性模型: 串联模型 (100%)
验证方法: 定时截尾 (100%)
环境条件: 机载振动 GJB 150.16A (100%)

术语偏好:
  可靠性预计 -> 内部称"可靠性摸底"
  大纲 -> 内部称"可靠性保证大纲"
```

#### 8.3 高稳定性段落识别

```
【高稳定性段落 -- 可自动复用】

可靠性大纲 . 第2章 "引用文件" -> 复用率 98%
  5个项目中该章节改动率<2%
  -> 新项目直接用，不需AI重写

降额报告 . "电阻降额"部分 -> 复用率 95%
  同平台产品基本不变
  -> 标记为"可直接继承"
```

#### 8.4 同平台项目克隆

```

  检测到"XXX-2型"与"XXX-1型"为同平台产品。
  建议从"XXX-1型"克隆文档架构。

  文档策划清单（16/18份文档相同）
  可靠性大纲（除产品名/指标外一致）
  环境试验大纲（完全一致）
  降额设计报告（元器件相同）
  FMECA（需更新故障分析）
  主数据（不同合同，需重新提取）

  克隆后自动更新产品名和批次信息。开始？
```

#### 8.5 与对话式写作的融合

## 五、技术实现要点

### 5.1 Prompt 模板系统改造

- 模板从"自然语言"改为"结构化定义"
- 增加变量注入机制（主数据、企业基线、历史决策）
- 每章输出 AI_META 注释（机器可读）
- 生成后自动解析 AI_META 驱动审查报告

### 5.2 对话引擎

- 基于 SSE 流式对话
- 左侧实时文档预览（markdown 渲染）
- 对话分阶段（策划->大纲->逐章->审查）
- 决策数据持久化（JSON 格式，供学习使用）

### 5.3 差异分析引擎

- 终稿提交时自动触发
- 五维分层对比（结构/指标/内容/格式/增删）
- 对比结果驱动模板和基线更新

### 5.4 依赖关系引擎

- 文档依赖用 YAML/JSON 结构化定义
- 前置检查在文档生成前自动运行
- 硬依赖（阻断）/ 软依赖（警告）/ 冲突检查（报警）
- 工作流导航动态计算可操作项

### 5.5 企业基线系统

- 按企业->产品类型->文档类型 三级存储
- 自动从终稿和历史决策中提取

## 十二、产品定位澄清：电子装备系统 vs 平台本体

### 12.1 领域边界

本系统服务于**军用电子装备系统**的研制文档编写，即搭载于各武器平台上的电子设备/子系统。不涉及平台本体（弹体、机体、船体、车体）的制造文档。

```
平台本体制造                  电子装备系统（本系统领域）
═══════════                  ══════════════════════════
弹载平台（导弹弹体）      →   弹载电子设备（导引头、飞控计算机、引信）
航空平台（飞机机体）      →   机载电子设备（雷达、航电、电子战吊舱）
舰船平台（船体结构）      →   舰载电子设备（指控系统、通信、导航雷达）
车载平台（车辆底盘）      →   车载电子设备（火控、侦察、通信终端）
```

**核心：同一套电子可靠性标准体系，差异在使用环境。**

### 12.2 环境参数来源

**环境参数来自用户下达的技术要求文件**（合同附件中的技术规格书、研制总要求等），不是根据平台推测。

```
环境参数获取流程：

  用户上传技术要求文件
         │
         v
  系统 AI 提取环境参数：
    - 工作温度范围
    - 振动条件（随机/正弦、量级、频谱）
    - 冲击条件
    - 湿度要求
    - 盐雾要求（如有）
    - 电磁兼容要求
    - 其他环境适应性要求
         │
         v
  提取结果确认（用户审核/补充）
         │
         v
  注入到文档生成流程
```

与主数据提取的关系：环境参数是主数据的一部分，在"主数据提取"环节一并处理。

### 12.3 平台选择的作用

平台选择不决定环境参数的具体数值，但影响以下方面：

| 平台 | GJB 299D 环境类别（πE） | 典型文档特征 |
|------|------------------------|-------------|
| 弹载 | ML（导弹发射） | 一次性使用、高可靠性、发射冲击 |
| 机载 | A（机载） | 持续运行、温度循环、适航要求 |
| 舰载 | N（舰载） | 长时持续、盐雾、密集电磁环境 |
| 车载 | G（地面） | 路面振动、宽温、机动性 |

> πE 环境系数来自 GJB/Z 299D 标准，系统预置。用于可靠性预计计算时的应力分析法公式：
> λp = λb × πE × πQ × πT × ...

### 12.4 299D 计算的完整数据来源

| 参数 | 来源 | 示例 |
|------|------|------|
| λb（基本失效率） | GJB/Z 299D 数据库（系统预置） | 查表获得 |
| πE（环境系数） | 平台选择 → 系统查表 | ML/A/N/G |
| πQ（质量系数） | 元器件清单（用户提供） | 军品级/工业级 |
| πT（温度系数） | 技术要求 + 元器件结温计算 | 结温105°C |
| πS（电应力系数） | 电路设计参数（用户提供） | S=0.5 |
| 环境参数 | 技术要求文件 AI 提取 | 温度、振动等 |

> 明确：只有 πE 由平台决定，其他参数均来自用户提供的文件和数据。

### 12.5 架构设计：一套模板 + 主数据注入 + 平台参数化

不需要为每个平台做一套独立模板。系统采用"通用电子模板 + 主数据注入 + 平台影响因子"模式：

```
项目创建流程：
  步骤3：产品信息
    - 领域：电子信息（系统默认）
    - 平台：弹载 / 机载 / 舰载 / 车载
    - 装备类型：雷达 / 通信 / 导航 / 电子战 / 指挥控制 / 其他
  步骤4：上传输入文件
    - 合同/任务书 → 主数据提取
    - 技术规格书/研制总要求 → 环境参数提取  ← 环境参数从这里来

生成文档时：
  文档模板（电子信息通用）
    + 主数据（从合同提取：产品名称、可靠性指标等）
    + 环境参数（从技术要求提取：温度、振动、湿度等）
    + 平台因子（系统根据平台选择：299D πE、典型文档推荐）
    → AI 生成时所有参数已锁定，不瞎编
```

### 12.6 不同平台可能产生的文档差异

虽然模板通用，但某些平台会产生额外要求。这些通过技术要求的实际内容决定，而非平台自动生成：

| 平台 | 技术要求中可能出现的额外文档/内容 |
|------|-------------------------------|
| 弹载 | 发射冲击环境适应性分析、一次性使用可靠性评估 |
| 机载 | DO-178/254 适配（若有适航要求）、高海拔环境 |
| 舰载 | 盐雾防护设计、电磁兼容性（舰船密集电磁环境） |
| 车载 | 越野运输振动、宽温工作 |

> 首版先做通用模板 + 主数据注入。平台特有文档通过"企业自定义模板"功能扩展。

### 12.7 对外表述建议

在演示和文档中，系统定位表述为：

> "面向军用电子装备研制过程的文档策划与编制系统。
>  覆盖弹载、机载、舰载、车载等武器平台的电子设备。"

以下能力在原需求讨论中识别但未纳入需求文档，属于系统构建的必要组成部分。

### 9.1 模板定制界面

**问题**：企业文档结构偶尔会微调，但不能让用户改代码。

**方案**：可视化模板编辑器
- 看到标准模板的章节结构
- 可以增加/删除/调整章节
- 可以修改章节约束（字数、必填字段）
- 保存为"企业自定义模板"
- 系统优先用企业模板，保留回退到标准模板的能力

### 9.2 项目创建向导

**问题**：新项目开始，用户不知道该做什么。

**方案**：六步引导
```
步骤1：项目基本信息（名称、代号、类型）
步骤2：研制阶段（方案阶段/工程研制/定型阶段）
步骤3：产品信息（类别：弹载/车载/舰载/...）
步骤4：上传输入文件（合同、任务书、技术规格书）
步骤5：系统自动规划文档清单 -> 确认/调整
步骤6：开始工作
```

### 9.3 GJB 格式导出/打印

**问题**：军工文档最终要打印成纸质文件存档。

**方案**：
- GJB 格式导出（A4、页边距、标题字体、表格样式）
- 封面自动生成（项目代号、文档名称、编制单位、日期）
- 审批页（两师签字栏：设计师、总师/副总师）
- 密级水印叠加（内部/秘密/机密）
- 支持导出为 Word（可再编辑）和 PDF（防篡改）

### 9.4 历史文档导入

**问题**：用户有几十上百份历史 Word/PDF 文档，不能浪费。

**方案**：
- 支持 Word/PDF 拖拽批量导入
- AI 自动拆分章节（基于标题编号识别）
- 提取可复用段落（去掉项目特定数据后的"模板化段落"）
- 自动建立企业基线（第一次导入即构建）
- 导入质量报告：哪些段落可复用、哪些文档有缺项

### 9.5 权限体系

```
编写员：生成文档、编辑自己的文档
校对员：审阅文档、添加批注
审核员：通过/驳回文档
批准员：最终批准
项目管理员：创建项目、分配任务、查看全部进度
系统管理员：模板管理、用户管理、系统配置
```

### 9.6 审计日志

体系审查时要追查：
- 谁在什么时间改了哪个文档的哪一部分
- 初稿->终版的完整进化链路
- AI 生成时的决策记录（提示词、参数、AI_META）
- 审批流转记录（提交时间、审批意见、通过/驳回）
- 登录和操作日志

> `ai_audit_log` 表已有基础，需扩展到覆盖所有文档操作。
