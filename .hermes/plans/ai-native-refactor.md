## 二、多人协同设计

### 2.1 项目工作空间

```
项目: XXX卫星导航抗干扰装置
│
├── 成员: 峰哥(项目经理/审批) 小王(质量师/编写) 老李(软件负责人/编写)
│
├── 阶段: 方案阶段
│   │
│   ├── 标准化大纲       ▸ 小王正在编写 (锁定中, 最后保存14:32)
│   ├── 质量保证大纲     ▸ 峰哥已审批 ✓ (版本V3, 14:15)
│   ├── 系统规范A类       ▸ AI生成中...(42%)
│   ├── 研制规范B类       ▸ 待编写 (等待上游: 系统规范A类)
│   └── 研制方案          ▸ 老李已提交审批 (待峰哥审批)
│
├── 通知: [上游就绪] 系统规范A类已完成, 研制规范B类可开始编写
│         [审批提醒] 老李提交了研制方案审批
│         [AI完成] 质量保证大纲第9章重新生成完毕
```

### 2.2 文档锁定 + 冲突检测

```sql
-- 文档编辑锁表
CREATE TABLE doc_edit_lock (
    id BIGSERIAL PRIMARY KEY,
    doc_ledger_id BIGINT NOT NULL,
    locked_by BIGINT NOT NULL,        -- 锁定用户
    locked_at TIMESTAMP NOT NULL,
    lock_type VARCHAR(16),            -- MANUAL_EDIT / AI_GENERATE / REVIEW
    heartbeat_at TIMESTAMP,           -- 心跳时间(检测死锁)
    UNIQUE(doc_ledger_id)             -- 一个文档一把锁
);
```

- 用户打开编辑→自动加锁 (30分钟心跳)
- AI生成某文档→自动加锁 (生成完成释放)
- 他人尝试编辑→提示"XX正在编辑/AI生成中，可查看只读副本"

### 2.3 审批工作流

```
编写人提交 → 项目经理审批 → 
  ├── 通过 → 文档状态=APPROVED, 通知下游文档可开始
  └── 驳回 → 文档状态=DRAFT, 带审批意见, 通知编写人
```

审批通过触发下游文档状态变更:

```
系统规范A类 审批通过
  → 研制规范B类 状态: 待编写(上游就绪) 
  → 详细设计 状态: 待编写(上游就绪)
  → 通知: 老李 "B类规范上游文档已就绪"
```

## 三、文档依赖链 (强化现有拓扑排序)

### 3.1 依赖类型

```sql
-- 文档依赖表 (在现有 doc_input_reference 基础上增强)
ALTER TABLE doc_input_reference ADD COLUMN IF NOT EXISTS dependency_type VARCHAR(32);
-- 取值: 
--   STRICT    = 上游必须审批通过才能开始
--   REFERENCE = 上游可参考, 不必等待
--   DATA      = 上游提供数据输入(主数据字段级)
```

### 3.2 依赖状态机

```
                    ┌─────────────────────┐
                    │  上游文档: 待编写     │
                    └─────────┬───────────┘
                              │ AI生成/人工编写
                    ┌─────────▼───────────┐
                    │  上游文档: 已编写     │
                    │  (下游: 等待中)      │
                    └─────────┬───────────┘
                              │ 提交审批
                    ┌─────────▼───────────┐
                    │  上游文档: 审批通过   │ ──→ 触发下游文档状态变更为"上游就绪"
                    │  → 通知下游编写人     │
                    └─────────────────────┘
```

### 3.3 Writer Agent 注入上游内容

生成下游文档时, 自动将已审批的上游文档摘要注入上下文:

```
Writer生成"研制规范B类"时:
  → 注入系统规范A类摘要: "系统由天线阵、接收机、电源、接口4大模块组成..."
  → 注入标准化大纲关键要求: "三化系数≥60%, 标准贯彻率≥95%"
  → Writer据此生成协调一致的内容
```

## 四、项目名称模糊匹配

### 4.1 三层匹配策略

```
用户输入: "北斗抗干扰装置方案阶段"
    │
    ├── 1. 精确匹配: project_code = "BD-KGR-001"
    │     └── 命中: 直接关联
    │
    ├── 2. 向量匹配: embedding(输入) vs embedding(project_name + aliases)
    │     └── 命中: "XXX卫星导航抗干扰装置" (余弦相似度 0.87)
    │     └── 提示: "您说的是「XXX卫星导航抗干扰装置」吗?"
    │
    └── 3. 关键词匹配: 提取"抗干扰""卫星""导航"→搜索项目标签
          └── 命中: 列出2个候选项目
```

### 4.2 项目别名表

```sql
CREATE TABLE project_alias (
    id BIGSERIAL PRIMARY KEY,
    project_id BIGINT NOT NULL,
    alias VARCHAR(256) NOT NULL,       -- "北斗抗干扰装置"
    alias_type VARCHAR(32),            -- USER_INPUT / SYSTEM_DERIVED / OFFICIAL_ABBR
    created_by BIGINT,
    usage_count INT DEFAULT 1,         -- 使用次数(越多越优先匹配)
    embedding VECTOR(768),             -- 向量化别名
    created_at TIMESTAMP DEFAULT NOW()
);
```

用户每次用不同名称提到项目→自动记录为别名→下次匹配更准。

## 五、五Agent架构 (70B加持)

### 5.1 合并 Orchestrator + Planner 为一个 70B Agent

70B模型(qwen2.5:72b)推理能力足够同时处理对话理解和文档规划, 不需要拆成两个Agent:

```
          ┌─────────────────────────────────────┐
          │    Orchestrator-Planner (72B)         │
          │                                       │
          │  · 自然语言理解                         │
          │  · 项目类型推断 → 阶段判定               │
          │  · 项目名称模糊匹配                      │
          │  · 文档清单规划(含上下游依赖)              │
          │  · 任务分解→分派Writer/Reviewer          │
          │  · 结果汇总→用户呈现                     │
          │  · 追问处理→调GJB Expert溯源             │
          └─────────────────────────────────────┘
```

**System Prompt 知识注入**:
- 7个研制阶段的定义和各阶段文档清单
- 15大文档类别及典型文档
- GJB 2993/6387/438B/1362A 等核心标准的文档要求摘要
- 文档间典型依赖关系(系统规范→研制规范→详细设计→产品规范)

### 5.2 其他Agent不变

| Agent | 模型 | 职责 |
|-------|------|------|
| GJB Expert | nomic-embed (RAG) | 按文档类型+章节检索GJB标准条款 |
| Writer ×4 | qwen2.5:7b ×4 | 并行逐章生成正文 |
| Reviewer | qwen2.5:7b | 逐章合规审查 |
| Memory | 代码(非LLM) | 记忆存取/流转 |

### 5.3 70B部署选项

| 方案 | 配置 | 并发生成速度 | 成本 |
|------|------|------------|------|
| Ollama GGUF q4_K_M | 单RTX 4090 24GB ×2 (层分割) | 单文档3-5分钟 | ~5万 |
| vLLM AWQ 4bit | 单A100 40GB | 单文档2-3分钟, 支持3并发 | ~8万 |
| vLLM AWQ 4bit | 双RTX 6000 Ada 48GB | 单文档2-3分钟, 支持5并发 | ~10万 |

**推荐**: 双RTX 4090 + Ollama, 性价比最高。72B q4约40GB, 两张4090共48GB显存刚好。

## 六、文档质量闭环 (防幻觉 + 持续提升)

### 6.1 生成时的四层防护

```
┌─────────────────────────────────────────────┐
│ Layer 1: 标准锚定                            │
│ GJB Expert RAG检索 → 注入当前章相关标准条款原文  │
│ Writer按条款要求逐项产出, 标注GJB条款引用        │
├─────────────────────────────────────────────┤
│ Layer 2: 结构化约束                           │
│ Writer输出格式: JSON → {"paragraphs":[...],   │
│ "tables":[...], "lists":[...]}, 禁止标题      │
│ 数值型指标: 只能来自主数据 或用 XXX占位         │
├─────────────────────────────────────────────┤
│ Layer 3: 交叉审查                             │
│ Reviewer逐章比对 GJB条款项→产出项, 打分       │
│ CRITICAL缺失→退回Writer重写(带审查意见)        │
├─────────────────────────────────────────────┤
│ Layer 4: 人工确认                             │
│ 连续2次Revision不通过→提给人工                 │
│ 用户可直接修改→存Correction Memory             │
└─────────────────────────────────────────────┘
```

### 6.2 质量评分卡

Reviewer输出每章评分:

```json
{
  "chapter": "3.2 功能特性",
  "scores": {
    "标准覆盖率": 0.85,    // GJB要求的条目, AI写了多少
    "数据完整性": 0.70,    // 应该填的主数据, 有多少是XXX
    "术语规范性": 0.95,    // 是否使用了GJB规范术语
    "格式正确性": 1.0,     // 段落/表格/列表格式是否正确
    "一致性": 0.80         // 与其他章节有无矛盾
  },
  "overall": 0.86,
  "issues": [
    {"severity": "MAJOR", "item": "缺少测速精度指标", "gjb_ref": "GJB 6387 3.4.1.c"}
  ]
}
```

```
文档: 质量保证大纲  [状态: 待编写]
```

### 7.3 模板的新角色

现有 `doc_template_v2` + `doc_template_chapter` 不再是生成强依赖, 而是:

- **AI参考**: Writer生成时注入"推荐章节结构", 但AI可以增减
- **学习材料**: 用户批准的内容→自动存入Learned Templates
- **质量基准**: Reviewer用模板作为"期望结构"比对AI产出
- **快速模式**: 有模板→跳过GJB Expert RAG(省token)

## 九、关键决策记录

| 决策 | 理由 |
|------|------|
| 管理层保留Java, AI引擎用Python | 管理层已有成熟代码, Python生态对LLM/RAG最友好 |
| Orch+Planner合并为72B单Agent | 72B推理能力足够, 减少Agent间通信开销 |
| Writer用7B×4而非72B | 7B足够写正文, 并行4个比串行72B更快且显存友好 |
| 模板体系降级但不删除 | 现有模板是宝贵知识资产, 用作AI参考和快速通道 |
| 不做微调, 用RAG+Prompt | 内网持续有新标准更新, RAG更灵活; 微调需要持续标注 |
| Ollama而非vLLM | 内网部署简单, 单机可运行, GGUF量化成熟 |
